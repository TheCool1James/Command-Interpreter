    The Parser, Compiler and Stuff

    These are the two key components to the whole process of lwlC → CbIL → Command Blocks. The parser first converts lwlC into CbIL where the compiler then compiles the list of instructions into valid command sequences. This article aims to document and detail the entire process of transforming the code.

    The Preprocessor

    Before any parsing of the code can happen, a preprocessor must come along to compile a list of valid symbols within the project. It will load any dependencies and compile a symbols list for each of those. Once finished, it must also identify the main entry point of the code within the "default" project. Only once an entry point has been found, can the parser come along to do its job. The entry point method must be static, and be inside a static class.

    The Parser

    Starting at the top of the entry point method, the parser makes its way to the bottom of the method. It will track down all the objects used, and their respective symbols, parse them, and add it to the final bytecode. The parser keeps a list of variables that it has initiated for GC (garbage collection) purposes. Every time it has to enter a referenced method (i.e., constructor etc.) or a code block (i.e., if statements), the code depth will be kept. If a variable's code depth exceeds that of the current code depth, that variable is considered to be garbage and will be cleaned up to free up space. The garbage collection is done during the bytecode → command stage of the process.
    For a more in-depth view on how the compiler treats each code statement, please refer to the "Language" portion of the documentation.

    Optimizations

    Due to the woefully unoptimized code that the parser may generate, the code needs to be cleaned up. Some optimizations take place during the parsing stage, and others, during the compilation stage. Here is a quick overview of what is provided:

    • Unrolling Loops: All loops will be unrolled while parsing the code.
    • Garbage Collection: Garbage will be dealt with during the compilation.
    • Dead Code Elimination: Dead code elimination is dealt with during parsing
      • Unused variables are removed during the parsing stage
      • Unreachable code is removed during the parsing stage
    • Data flow analysis: Some data may be collapsed: { b = 1; a = b + 2; } → { a = 3; }
